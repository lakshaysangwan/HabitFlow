# HabitFlow

A full-stack habit tracker built on Cloudflare's edge infrastructure. No servers, no containers — everything runs at the edge.

**Live:** https://habitflow.uk

---

## Stack

| Layer | Technology |
|-------|-----------|
| Frontend | React 18, React Router, Tailwind CSS, Radix UI, Recharts |
| Backend | Hono (running inside Cloudflare Pages Functions) |
| Database | Cloudflare D1 (SQLite at the edge) |
| ORM | Drizzle ORM |
| Auth | JWT (HS256 via `jose`) stored in httpOnly cookies |
| Hosting | Cloudflare Pages |
| Tests | Vitest + `@cloudflare/vitest-pool-workers` (real Miniflare runtime) |

## How FE and BE communicate

The frontend and backend are served from **the same domain**. There is no separate API server.

- Vite builds the React app to `./dist`
- Cloudflare Pages serves `./dist` as static files
- Any request to `/api/*` is intercepted by Pages Functions before reaching static files
- The frontend calls `fetch('/api/...')` — same-origin, no CORS needed in production

```
Browser
  │
  ├── GET /         → serves dist/index.html  (static)
  ├── GET /assets/  → serves dist/assets/*    (static)
  └── ANY /api/*    → functions/_middleware.ts → functions/api/[[catchall]].ts → Hono
```

---

## Authentication flow

1. `POST /api/auth/login` — validates credentials, returns a signed JWT in an `httpOnly; SameSite=Strict` cookie
2. Every subsequent `/api/*` request — `_middleware.ts` reads the cookie, verifies the JWT, checks `token_version` against DB (for revocation), attaches `userId` to context
3. Logout / password change — increments `token_version` in DB, invalidating all existing tokens
4. Sliding-window refresh — if a valid token expires in < 2 hours, middleware silently issues a new one

**Registration is invite-only.** An admin must create an invite code first.

## Database schema

```
users           — id, username, password_hash, display_name, timezone, is_god, token_version
invite_codes    — code, max_uses, current_uses, created_by
tasks           — id, user_id, name, color, frequency_type, frequency_days, data_type, status, sort_order
completions     — task_id, user_id, completed_date (YYYY-MM-DD), data_text, data_number
audit_log       — event_type, user_id, ip_address, user_agent, metadata
rate_limits     — key (e.g. "auth:1.2.3.4"), count, window_start
```

D1 is SQLite. All dates stored as ISO strings. `completed_date` is stored in the user's local timezone (sent by the client).

## Local development

### Prerequisites
- Node.js 20+
- Wrangler CLI (`npm install` installs it locally)
- Cloudflare account with D1 access (for remote DB commands)

### Setup

```bash
npm install
```

Create `.dev.vars` in the project root (gitignored):
```
JWT_SECRET=any-string-at-least-32-chars-long
ENVIRONMENT=preview
ALLOWED_ORIGIN=http://localhost:5173
```

Apply migrations to local D1:
```bash
npm run db:migrate:local
```

### Running locally

You need two terminals:

**Terminal 1 — Frontend (Vite dev server on :5173):**
```bash
npm run dev
```

**Terminal 2 — Backend (Wrangler Pages dev on :8788):**
```bash
npm run worker:dev
```

The Vite dev server proxies `/api/*` requests to `:8788` (configured in `vite.config.ts`). Open http://localhost:5173.

## Testing

Tests run in **Miniflare** (local Cloudflare Workers runtime) — they hit a real in-memory D1, not mocks.

```bash
npm test           # run all tests once
npm run test:watch # watch mode
```

Test environment vars are injected via `vitest.config.ts` (miniflare.vars). In CI, `.dev.vars` is created by the workflow before tests run.

## Environment variables

| Variable | Where | Description |
|----------|-------|-------------|
| `JWT_SECRET` | CF dashboard (Secret) / `.dev.vars` locally | HS256 signing key — minimum 32 chars. **Never commit.** |
| `ENVIRONMENT` | `wrangler.toml` | `production` or `preview` — controls CORS and cookie Secure flag |
| `ALLOWED_ORIGIN` | `wrangler.toml` | Production CORS origin. Only used for cross-origin requests (rare). |
| `DB` | `wrangler.toml` (binding) | D1 database — injected by Cloudflare, not a string |

`JWT_SECRET` **must** be set in the Cloudflare Pages dashboard under Settings → Environment variables → Add secret. It is not in `wrangler.toml`.

## Deployment

Deployment is automatic via Cloudflare Pages Git integration:

| Branch | Environment | Database |
|--------|-------------|----------|
| `main` | Production → `habitflow-5ve.pages.dev` | `habitflow-db` |
| any other branch | Preview → unique `*.habitflow-5ve.pages.dev` URL | `habitflow-db-preview` |

There is no manual deploy command needed. Push to `main` → Pages builds and deploys.

The GitHub Actions workflow (`.github/workflows/test.yml`) runs tests and build on every push — it is separate from the Cloudflare Pages deployment.

### First-time production setup

After deploying for the first time, apply migrations and seed an admin user:

```bash
# 1. Apply migrations to production DB
npm run db:migrate:remote

# 2. Apply migrations to preview DB
npx wrangler d1 migrations apply habitflow-db-preview --remote --env preview

# 3. Create an invite code directly in D1
npx wrangler d1 execute habitflow-db --remote --command \
  "INSERT INTO invite_codes (id, code, max_uses) VALUES (lower(hex(randomblob(16))), 'ADMIN_SETUP', 1)"

# 4. Register your admin account via the app using code ADMIN_SETUP

# 5. Promote the account to god-mode
npx wrangler d1 execute habitflow-db --remote --command \
  "UPDATE users SET is_god = 1 WHERE username = 'your_username'"

# 6. Delete the bootstrap invite code
npx wrangler d1 execute habitflow-db --remote --command \
  "DELETE FROM invite_codes WHERE code = 'ADMIN_SETUP'"
```

## Key design decisions

**Why Cloudflare Pages Functions instead of a separate Worker?**
Single deployment unit. FE and BE ship together, share the same domain, zero CORS complexity in production.

**Why D1 for rate limiting instead of KV or Durable Objects?**
Simplicity. D1 is already the primary store. Rate limit records are tiny and the SQLite upsert pattern works fine at this scale.

**Why httpOnly cookies instead of Authorization headers?**
XSS-resistant. The token is never accessible to JavaScript. `SameSite=Strict` adds CSRF protection.

**Why `token_version` in the users table?**
Allows instant token revocation without a token denylist. Logout and password change both increment `token_version`, invalidating all existing sessions server-side.

**Why store `completed_date` as a string in the user's timezone?**
Habit tracking is inherently local-time. A completion at 11pm in IST should count for that day in IST, not UTC the next day.
